# Google Play 上架流程
> 新包上架 + 老包迭代完整经验 | 2026-04-15 | #发布 #GP #运维

## 🎯 两大场景

| 场景 | 核心区别 |
|------|----------|
| 新包上架 | 需注册账号、准备全套素材、配置GP/Firebase连接 |
| 老包迭代 | 仅需传新包、更新版本说明，无需重复填写商店信息 |

## 🔧 新包上架完整流程

### 阶段一：开发侧准备

| 事项 | 说明 |
|------|------|
| 包名 | 全局唯一，一旦确定不可更改 |
| 应用图标 | 适配各分辨率，符合GP规范 |
| 环境/风格 | 区分生产/测试环境配置 |
| google-service.json | 绑定GP服务（如推送、统计） |
| Firebase连接 | 若需Firebase，在Firebase控制台与GP建立关联 |

### 阶段二：市场侧准备

| 事项 | 说明 |
|------|------|
| 注册账号 | GP开发者账号，一次性$25 |
| 准备素材 | 图标、截图、功能图、视频、描述、隐私政策 |

### 阶段三：内测（Internal Testing）

**特点**：
- 仅测试列表中的用户可通过商店链接下载
- 审核快（几小时）
- 优先传 **AAB** 格式（GP会为不同设备生成优化APK）

**操作步骤**：
1. GP Console → Internal Testing → 上传AAB
2. 等待审核通过
3. How testers join your test → Copy Link
4. 添加测试账号列表

**常见障碍**：
> 只发布了internal testing，显示unreviewed。通过链接访问有“接受成为测试者”面板，但点击“Download it from Google Play”后显示Item Not Found。
> 
> **原因**：审核尚未完全生效，或账号未加入测试列表。等待审核完成 + 确认账号已添加。

### 阶段四：封闭测试（Closed Testing）

| 要点 | 说明 |
|------|------|
| 个人开发者 | **强制**：20人 + 连续14天真实测试 |
| 企业开发者 | 不强制 |
| 目的 | 收集真实用户反馈，满足GP政策要求 |

### 阶段五：公测（Open Testing）

| 要点 | 说明 |
|------|------|
| 用户范围 | 任何用户均可通过商店链接下载 |
| 审核时长 | 约1-2天 |
| 用途 | 扩大测试范围，验证稳定性 |

### 阶段六：正式发布

**关键配置**：

| 配置项 | 说明 |
|------|------|
| 包格式 | **必须用AAB** |
| 国家/地区 | 选择发布范围 |
| 内容分级 | 填写问卷，获取分级结果 |
| 数据安全表单 | 声明数据收集和使用方式 |
| 应用签名 | 自行管理 或 GP管理（推荐GP托管） |
| 分阶段发布 | 支持灰度发布（如10% → 50% → 100%） |

**审核时长**：1-7天

### 阶段七：审核被拒处理

1. 查看GP Console中的审核反馈
2. 根据反馈修改代码/配置
3. 停用当前被拒版本
4. 提交新版本重新审核

## 🔧 老包迭代流程

| 步骤 | 说明 |
|------|------|
| 1 | 修改versionCode（唯一递增） |
| 2 | 打包新AAB |
| 3 | GP Console → 选择应用 → Production / Testing渠道 |
| 4 | 上传新包，填写版本更新说明 |
| 5 | 提交审核 |

**注意**：不需要重新填写商店列表、内容分级、数据安全表单等（除非有变更）。

## 📝 核心概念

| 概念 | 说明 |
|------|------|
| versionCode | 系统识别内部版本标识，必须唯一递增 |
| AAB | Android App Bundle，GP推荐格式，可动态生成优化APK |
| 分阶段发布 | 灰度发布，逐步放量，降低风险 |

## ⚠️ 注意事项
1. 包名一旦确定无法修改，立项时谨慎选择
2. 签名密钥（.jks）务必备份，丢失无法更新应用
3. 个人开发者必须完成封闭测试（20人/14天）才能申请正式发布
4. Internal Testing的“Item Not Found”通常是审核未完成，耐心等待
5. 正式发布前确保数据安全表单和内容分级已填写完整
